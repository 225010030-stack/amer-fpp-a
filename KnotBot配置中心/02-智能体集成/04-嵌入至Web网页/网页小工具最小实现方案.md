# 网页小工具最小实现方案（按钮式，低 Token）

> 目标：同事打开内网页面，点按钮即可运行检查与生成文件，无需终端。  
> 定位：这是“脚本触发器 + 文件下载器”，不是聊天AI。

---

## 1. 最小功能范围（MVP）

页面只做 3 个按钮：
- `总检查`（目录预检 + HC校验）
- `目录预检`
- `生成提单文本块`

页面只显示 3 类结果：
- 执行状态（成功/失败）
- 执行摘要（耗时、条数）
- 输出文件列表（可点击下载/打开）

---

## 2. 前后端接口设计

## 前端页面

一个简单页面即可：
- 左侧：按钮区
- 右侧：最近执行记录
- 底部：文件列表

## 后端接口（HTTP）

1) 触发执行  
- `POST /api/run`
- body 示例：

```json
{
  "action": "run_full_check",
  "root": "/Users/zhangwenjing/Desktop/工作提效"
}
```

2) 查询最近结果  
- `GET /api/jobs/latest`

3) 下载文件  
- `GET /api/files?path=<encoded-path>`

4) 健康检查  
- `GET /api/health`

## 4. 部署方式建议

## 方式A：单机部署（最快）

- 在你们共享可访问的一台 Mac/服务器上跑后端服务
- 前端用静态页面（nginx）或后端模板页面
- 适合先试点

## 方式B：内网容器部署（推荐）

- 后端封装 Docker
- 挂载 `工作提效` 目录
- 用公司内网网关发布
- 适合长期维护和多同事使用

---

## 5. 权限与安全建议（必须）

- 只允许白名单 action，不允许任意命令执行
- 文件下载只允许 `工作提效` 根目录内的路径
- 每次执行记录操作人、时间、action、输出文件
- 增加简单鉴权（企业单点或固定 token）
- 给“写操作”留开关，默认只读/只生成文件

---

## 6. 错误处理策略

后端统一返回：
- `ok`：是否成功
- `error_code`：错误码
- `message`：可读错误信息
- `suggestion`：下一步建议
- `log_file`：错误日志文件路径

错误日志统一落地：
- `工作提效/运行日志/错误日志_YYYYMMDD_HHMMSS.md`

---

## 7. 最小技术栈建议

后端：
- Python FastAPI（轻量、易维护）

前端：
- 原生 HTML + JS（先不要上复杂框架）

原因：
- 目标是“快上线、稳运行”，不是做产品级前端工程。

## 10. 为什么这个方案适合你们现在

- 不依赖同事 token 额度
- 复用你已经做好的脚本和 Skill
- 输出全是文件，便于审计、交接、复盘
- 后续再加 AI 也容易（在现有接口上叠一层解释能力）
